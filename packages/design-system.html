uvr / component library
v0.1 sketch / python + rich

Build it from a small vocabulary of primitives.

Everything in the uvr terminal is composed of ~10 reusable components. Most map to a Rich primitive you'll style; two or three are worth writing yourself so the visual grammar (the bold-magenta header + hyphen rule, the ASCII progress bar) stays consistent across every command. Below: each primitive in isolation, its API shape, and what Rich gives you for free.

The big idea: wrap Rich's Console in your own uvr.ui module that exposes these components as functions. Every command (plan, release, pin) calls only your wrappers -- never Rich directly. That way the design stays consistent and you can swap Rich out later if needed.

Color language

One hero color, plus semantics. uvr uses exactly six terminal colors. The hero (bright_magenta) is the brand and only ever marks things the user types or follows. The rest are semantic -- each one means one thing, always, across every command. If a color doesn't carry meaning here, it doesn't ship.

The brand callout. Not good, not bad -- just look here. Use it for section titles, the commands a user types or copies, the package names being acted on, the progress-bar fill, the prompt cursor, the spinner frame. A name pointing at something stable -- a git ref, a tag, a baseline version, a step name in a pipeline (uvr-build). Cyan distinguishes "things the system tracks" from "things the user runs" (which are uv). One is a noun; the other is a verb.

structure / chrome only dim ---- / |---| / $ dim / ANSI 90 / #6b6357
Formatting only -- never contentful text. Hyphen rules under section headers, the empty cells of an ASCII progress bar, the / separator, the leading $ in a shell example. These are scaffolding the eye should skim past. Words, numbers, names, paths, timings, hints -- all default fg, never dim. If a reader has to read it, it's content; if they only need to feel its shape, it's chrome.

default fg is the workhorseMost words on screen are the terminal's plain foreground -- readable, full contrast. Color is the exception, applied only where it earns meaning.
color carries meaning twiceAlways pair color with a word (changed, error:, [ok]). When NO_COLOR strips ANSI, the meaning has to survive.
don't dim contentful textTimings, hints, badge labels, table headers, paths -- all default fg. Dim is only for the hyphens, the empty bar cells, the $ prompt -- chrome, never content.
one color, one meaningEach token owns its job: uv is callout, ok/warn/error are state, ref is identifier, dim is chrome. Don't reuse a color for a second purpose -- a "magenta path" or a "yellow command" makes both meanings unreadable.

01Section rich: Text + manual hyphen rule
Planning -------- Discovered 4 packages Resolved 4 baselines

What it is

The signature uvr move: bold magenta title, then a hyphen rule the same length on the next line, then body text indented two spaces. Used to open every distinct phase of a command.
def section(title: str) -> None: console.print(title, style="bold bright_magenta") console.print("-" * len(title), style="dim")

Why custom

Rich's Rule uses Unicode box characters by default. We want a literal hyphen, equal-length to the title -- that's what makes it look like --help output.
02ProgressLine rich: Progress w/ custom columns
|##########| Discovered 3 packages 9ms |##########| Resolved 3 baselines 57us |#####-----| Detected 2 changed 4ms |###-------| Computed versions 2ms |----------| Generated 2 release notes 625ns

What it is

Fixed-10-cell ASCII bar (|##---|), label, then a right-aligned timing in default fg. Use one per discrete unit of work; let them stack vertically inside a Section.
from rich.progress import Progress, BarColumn, TextColumn, TimeElapsedColumn bar = BarColumn( bar_width=10, complete_style="bright_magenta", finished_style="bright_magenta", ) with Progress( TextColumn(" |"), bar, TextColumn("|"), TextColumn("{task.description}"), TimeElapsedColumn(), console=console, transient=False, ) as p: t = p.add_task("Discovered 3 packages", total=3) # ... advance as work happens

Tip

Override BarColumn's glyphs to # and - via subclass -- Rich's default is Unicode blocks (#-). That single override is what makes uvr's bars look like uvr's bars.
03Table rich: Table(box=None, pad_edge=False)
STATUS PACKAGE VERSION PREVIOUS CHANGES COMMITS changed my-core 0.1.0 -> 0.2.0 0.1.0 feat,fix 12 changed my-auth 0.1.0 -> 0.1.1 0.1.0 fix 4 unchanged my-api 0.4.2 0.4.2 -- -- unchanged my-cli 1.0.0 1.0.0 -- --

What it is

Fixed-width column table, no borders, plain header row in caps. Padding is spaces only. Rich does column sizing for you -- just make sure to disable the box.
from rich.table import Table t = Table(box=None, pad_edge=False, show_edge=False, header_style="", padding=(0, 2)) t.add_column("STATUS") t.add_column("PACKAGE") t.add_column("VERSION") t.add_row(badge("changed"), "my-core", "0.1.0 -> [b]0.2.0[/b]") console.print(t)

Don't

Don't enable any box style. Even SIMPLE adds dividers that fight the hyphen-rule grammar.

05Confirm rich: Confirm.ask
Dispatch release? [y/N] _

What it is

The interactive yes/no prompt. Bold question, magenta [y/N] hint, magenta cursor. Rich's Confirm.ask handles the loop and parsing -- you only style the prompt string.
from rich.prompt import Confirm ok = Confirm.ask( "[bold]Dispatch release?[/bold] [bright_magenta]", default=False, console=console, ) if ok: dispatch()

Tip

Set default=False so a bare Enter aborts. Add --yes on the command to skip the prompt entirely (for CI).
06Pipeline custom / indent + arrow
run uvr-build run uvr-publish my-core -> pypi my-auth -> pypi my-api -> ghcr.io

What it is

Two-level indented list with dim -> arrows for routing. Use any time something flows from A to B (package -> registry, version -> version, baseline -> head).
def pipeline(steps: list[Step]) -> None: for s in steps: console.print(f" run [cyan]{s.name}[/cyan]") for route in s.routes: console.print( f" {route.src} [dim]->[/] {route.dst}" )
07ErrorBlock styled Text / no traceback panel
error: stale pin in my-auth/pyproject.toml expected my-core ^0.2 got my-core ^0.1 fix -> uvr pin --update -> git commit -am 'pin my-core'

What it is

Red error: prefix, one-line summary, indented body with expected/got, then a fix -> block of copy-pasteable next commands. Always end with the fix.
def error(summary: str, *, detail: dict[str, str] | None = None, fixes: list[str] | None = None) -> None: console.print(f"[bold red]error:[/] {summary}") if detail: w = max(map(len, detail)) for k, v in detail.items(): console.print(f" {k:<{w}} {v}") if fixes: console.print() for i, cmd in enumerate(fixes): lead = " fix" if i == 0 else " " console.print(f"{lead} [dim]->[/] {cmd}")

Don't

Don't use Rich's install_rich_traceback for expected errors -- that's for crashes. Tracebacks should only fire on uncaught exceptions; user-facing failures get this block.
08KV (aligned pairs) custom / format strings
baseline v0.4.1 strategy conventional branch main / 3 ahead of origin/main tagged 2d ago packages 4 (2 changed)

What it is

Two-column aligned key/value pairs. Keys are default fg, padded to the widest key. Used for status summaries and config display.
def kv(pairs: dict[str, str]) -> None: w = max(map(len, pairs)) for k, v in pairs.items(): console.print(f" {k:<{w}} {v}")
09Hint / Footer styled Text / dim
Next: uvr release to dispatch. See uvr pin --check to verify. Hint: pass -y to skip the prompt.

What it is

Trailing line that points at the next thing. Always include the literal command in magenta so the user can copy-paste it. Every successful command should end with a hint or a blank line.
def hint(text: str, cmd: str | None = None) -> None: if cmd: console.print(f" {text} [bright_magenta]{cmd}[/]") else: console.print(f" {text}")
10Banner custom / one line, no figlet
uvr 0.4.2 / a release planner for python monorepos type help for commands / uvr release for the demo

What it is

Two-line opener for uvr with no args, or --help. One bold word + a tagline. No ASCII art logo, no figlet, no color block -- those age badly and make the tool feel hobbyist.
def banner() -> None: console.print( "[bold bright_magenta]uvr[/] " f"{__version__} [dim]/[/] a release planner " "for python monorepos" ) console.print( "type [bright_magenta]help[/] for commands " "[dim]/[/] [bright_magenta]uvr release[/]" " for the demo" )
11Spinner rich: Status / custom frames
| Fetching baseline tags git ls-remote origin // frames cycle: | / - \ at ~80ms / Resolving from PyPI my-core, my-auth - Pushing tags to origin 2 tags \ Awaiting GitHub release workflow run #4821 [ok] Fetched baseline tags 214ms

What it is

For genuinely indeterminate work -- network calls, git fetch, polling a CI run. Single ASCII frame cycling | / - \ in magenta, label in default fg, optional default-fg subtitle. On completion, the frame is replaced with a green [ok] and an elapsed time. Not a substitute for ProgressLine -- use only when you can't measure progress.
from rich.status import Status # custom spinner registered once at import time from rich.spinner import SPINNERS SPINNERS["uvr"] = { "interval": 80, "frames": ["|", "/", "-", "\\\\"], } def spinner(label: str, subtitle: str | None = None">): text = label if not subtitle else ( f"{label} {subtitle}" ) return Status( text, console=console, spinner="uvr", spinner_style="bright_magenta", ) # usage -- context manager, auto-clears on exit with spinner("Fetching baseline tags", "git ls-remote origin") as s: tags = git.ls_remote() s.update("Resolving from PyPI") # swap label live pypi.fetch(tags) console.print(f"[green][ok][/] Fetched baseline tags {ms}ms")

When to reach for this vs. ProgressLine

  • ProgressLine when you know the unit count (4 packages, 2 baselines).
  • Spinner when the duration is opaque -- network, subprocess, polling.
  • Never both at once. A spinner inside a Section breaks the line-per-step rhythm; only one spinner is live at a time, and it sits between Sections.

Don't

Don't use Rich's default dots spinner (Unicode braille). It's beautiful but off-brand -- it can't appear inside a hyphen-rule grammar. Stick to ASCII | / - \ so the spinner can be screenshotted, pasted into a README code block, or rendered in a CI log without box-drawing artifacts.

Suggested project structure

uvr/ |--- __init__.py |--- cli.py // click / typer command tree |--- commands/ | |--- plan.py // each command thin -- calls planner + ui | |--- release.py | |--- pin.py | |--- bump.py | `--- status.py |--- planner.py // the actual logic. zero printing. |--- git.py |--- pypi.py `--- ui/ // EVERYTHING in this folder is the design system |--- __init__.py // re-exports section, table, badge, progress, ... |--- console.py // the singleton Console + theme |--- section.py // 01 |--- progress.py // 02 |--- table.py // 03 -- no-box wrapper around rich.table |--- badge.py // 04 |--- confirm.py // 05 |--- pipeline.py // 06 |--- error.py // 07 |--- kv.py // 08 |--- hint.py // 09 |--- banner.py // 10 `--- spinner.py // 11
Discipline: command modules import from uvr.ui only -- never from rich directly. If you find yourself reaching for rich.panel or rich.markdown from a command, that's a sign you need a new primitive in uvr.ui first. The whole point of this layer is that the design grammar lives in one place.
What I left out: no Panel / Box components, deliberately -- boxes fight the hyphen-rule grammar. If you ever need to set something apart, use a Section instead. Also no Markdown renderer; uvr never prints Markdown to a terminal (the README is for GitHub, the terminal is for plain text).
